| Поле | Значение |
| --- | --- |
| Цель | Согласовать подход к созданию и обновлению проектной документации |
| Аудитория | Разработчики, AI агент |
| Ответственный | Команда разработки |
| Последнее обновление | 2025-12-23 |
| Связанные документы | [Документация проекта](../README.md) |

## Принципы

- Перед началом работы переходи к `documentation/README.md` — там собрана актуальная карта разделов и ссылки на ключевые материалы.
- Документация должна быть полезна для планирования задач и ревью, поэтому фиксируй причинно-следственные связи и зависимые компоненты.
- Поддерживай единый стиль: русский язык, короткие абзацы, списки для правил и сценариев.
- Каждый файл начинается с таблицы метаданных (цель, аудитория, дата, ответственный, ссылки).

## Структура каталогов

- `architecture/` — ключевые архитектурные решения и слои приложения.
- `cards/` — документы о типах карточек, прогрессе и сценариях обучения.
- `data/` — описание базы данных, внешних источников, диаграммы.
- `process/` — регламенты и рабочие процессы (включая этот документ).

При добавлении нового направления создай подкаталог и добавь ссылку в `documentation/README.md`.

## Регламент обновления

- Обновляй дату и содержимое при каждом существенном изменении функциональности.
- В PR указывай, какие документы затронуты, чтобы не терялись связи.
- При большом изменении архитектуры подготовь обзорный раздел и приложи диаграммы/схемы.
